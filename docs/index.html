Shader Math Deep Dive

Mathematical foundations of voronoi_reactive.fs, walked through function-by-function from building blocks up to final composition.

1. Hash Functions

Both hash1 and hash2 map a 2D input to a pseudorandom output (scalar or 2-component) in [0, 1). They're the source of all randomness in the shader.

The construction, step by step

Take hash1:

vec3 p3 = fract(vec3(p.xyx) * 0.1031);       // (1) scale + fold
p3 += dot(p3, p3.yzx + 33.33);                // (2) mix via dot product
return fract((p3.x + p3.y) * p3.z);           // (3) collapse + fold

Step (1) — p.xyx promotes the 2D input to 3D by repeating x. The multiply by 0.1031 rescales into a fractional range, and fract() wraps it to [0, 1). This is the first “folding” — it maps the infinite integer lattice into a bounded domain.

Step (2) — The dot product against a permuted version of itself (p3.yzx) creates cross-component mixing. The + 33.33 offset ensures that even inputs near zero produce large intermediate values. This is the avalanche step — small input changes cascade into large output changes.

Step (3) — A final multiply-and-fract collapses back to a scalar. The multiplication is the key nonlinearity: it creates high-frequency variation that fract then wraps.

Why these constants?

The constants (0.1031, 0.1030, 0.0973, 33.33) are empirically tuned by Dave Hoskins. The design goals:

  • Irrational-ish ratios — the three scale factors are close but not equal. If they were identical, the swizzled components would be correlated. The slight differences break symmetry.
  • Large offset (33.33) — prevents the dot product from being near-zero for small inputs, which would kill the avalanche effect.
  • No special mathematical significance — unlike the golden ratio constants in some hashes. These were found by visual inspection of distribution quality on GPU.

Why “no sine”?

The classic GPU hash is fract(sin(dot(p, vec2(12.9898, 78.233))) * 43758.5453). The problems:

  1. Precision — sin() on large inputs depends on the GPU's trig implementation. Different GPUs compute sin(78432.7) differently. The high-frequency part of the sine curve amplifies these errors.
  2. Periodicity — sin is periodic. For large enough inputs the hash starts repeating in visible patterns.
  3. Speed — on some mobile GPUs, sin is a multi-cycle instruction. Multiply-and-fract is uniformly fast.

Hoskins' hash uses only fract, dot, +, and * — all of which are exact (or nearly exact) across GPU architectures. The result: portable, consistent randomness.

Distribution quality

This class of hash isn't cryptographically uniform, but it's visually uniform — the autocorrelation is low enough that human eyes can't detect patterns. Toggle between the two below to see the difference:

2D scatter of hash2 output for a 64×64 grid of integer inputs. Both look uniform at first glance — the no-sine version is portable across GPUs; the sine version may produce different patterns on different hardware.

hash2 differences

vec3 p3 = fract(vec3(p.xyx) * vec3(0.1031, 0.1030, 0.0973));

The three different scale constants mean the three intermediate components decorrelate faster. The output extraction also differs:

return fract((p3.xx + p3.yz) * p3.zy);

This produces two outputs from the same intermediate state by using different swizzle patterns. The two output components are not statistically independent (they share the same intermediate p3), but they're uncorrelated enough for Voronoi seed placement.

2. Voronoi / Worley Noise

F1 and F2: geometric meaning

The shader evaluates Voronoi on a grid. Each integer cell (i, j) gets a seed point (random position inside the cell). For any query point p:

  • F1 = distance to the nearest seed
  • F2 = distance to the second-nearest seed

Geometrically:

  • F1 defines the classic Voronoi diagram. The locus F1 = 0 is the set of seed points themselves. The Voronoi cell for seed s is the region closer to s than to any other seed.
  • F2 − F1 measures how close the query point is to a cell boundary. On the boundary itself, the two nearest seeds are equidistant, so F2 − F1 = 0. Deep inside a cell, F2 − F1 is large.

The key insight: F2 − F1 is a smooth scalar field whose zero set is exactly the Voronoi edge network. That's why it's so useful — you get edges “for free” from the distance computation, no derivative or finite-difference needed.

Toggle between the three modes to see what F1, F2, and F2−F1 actually look like. White dots are seed points.

Static Voronoi with visible seed points. F1 shows distance-to-nearest (bright near seeds). F2 shows distance-to-second-nearest. F2−F1 reveals the edge network — bright lines where two seeds are equidistant.

Why the 3×3 neighborhood search is sufficient

The loop checks a 3×3 grid of cells centered on the cell containing the query point. The seed for cell (i, j) is confined to [0.1, 0.9] within the cell (seedHash * 0.8 + 0.1). Drift can push it further — up to about ±0.35. Worst case:

seedBase + drift ∈ [0.1 - 0.35, 0.9 + 0.35] = [-0.25, 1.25]

So a seed can wander at most ~0.25 units outside its home cell. Seeds two cells away are at minimum 0.75 units from the cell boundary, while there's always a closer seed in the 3×3 block. So 3×3 is sufficient with comfortable margin.

Seed animation model

Each seed has two motion modes blended by driftChaos:

Circular drift:

float angle = 6.2832 * seedHash.x + animTime * (0.3 + seedHash.y * 0.7);
vec2 circularDrift = 0.35 * vec2(cos(angle), sin(angle));

Simple circular motion — (r·cos(ωt + φ), r·sin(ωt + φ)) — with phase φ = 2π · seedHash.x and angular velocity varying per cell in [0.3, 1.0]. Smooth, predictable orbits.

Chaotic drift: a keyframe random walk. Time is quantized into phases; at each boundary a new random target is generated. The seed smoothly interpolates between targets using Hermite smoothstep for C1 continuity — no visible snapping.

The blend: mix(circularDrift, chaoticDrift, driftChaos). Drag the slider below to scrub between pure circular orbits and pure random walk. Watch how the seed dots move.

At 0: smooth circular orbits. At 1: random walk with Hermite-interpolated keyframes. The default in the shader is 0.3 — mostly circular with some random perturbation.

3. Value Noise & Spatial Warp

Hermite interpolation: why 3t² − 2t³?

In valueNoise2:

f = f * f * (3.0 - 2.0 * f);   // f ← 3f² - 2f³

This is the Hermite basis function h(t) = 3t² − 2t³, mapping [0,1] → [0,1] with:

  • h(0) = 0, h(1) = 1 (interpolation)
  • h'(0) = 0, h'(1) = 0 (C1 continuity)

Why not linear? With linear interpolation between grid values, the derivative is discontinuous at integer lattice boundaries. You'd see visible seams. The Hermite smoothstep guarantees continuous first derivatives everywhere.

Why not 6t&sup5; − 15t&sup4; + 10t³ (Perlin's improved smoothstep)? That gives C2 continuity, which matters for lighting computations where you differentiate the noise to get normals. For a displacement warp, C1 is sufficient — the human eye doesn't detect second-derivative discontinuities in color fields.

Hover over the chart to compare the three curves:

Linear has sharp corners at 0 and 1. Hermite (blue) rounds them to zero slope. Quintic (orange) also zeroes the second derivative at the endpoints.

The noise itself

vec2 a = hash2(i) - 0.5;
vec2 b = hash2(i + vec2(1,0)) - 0.5;
vec2 c = hash2(i + vec2(0,1)) - 0.5;
vec2 d = hash2(i + vec2(1,1)) - 0.5;
return mix(mix(a, b, f.x), mix(c, d, f.x), f.y);

This is bilinear interpolation (with Hermite-smoothed coordinates) of random vectors at the four corners of the integer cell. The - 0.5 centering means the noise has zero mean — important for a warp displacement, so the average UV position doesn't shift.

Note: this is value noise, not gradient noise (Perlin noise). Value noise interpolates random values; gradient noise interpolates random gradients dotted with position offsets. Value noise has a slightly “blobby” character. For a spatial warp, this is fine — you want broad, smooth distortions.

The two-octave warp

vec2 warpOffset = valueNoise2(uv * 3.0 + TIME * 0.08);
warpOffset += valueNoise2(uv * 7.0 - TIME * 0.05) * 0.5;
uv += warpOffset * warp * 0.25;

The frequency ratio is 7/3 ≈ 2.33, and the amplitude ratio is 0.5. Close to standard fBm (lacunarity 2, persistence 0.5) but with slightly wider frequency spacing. The opposite time drift directions mean the two octaves slide past each other, creating a slowly evolving distortion.

The warp modifies UV before it enters the Voronoi computation. The cells themselves get distorted — boundaries curve and wobble. Drag the slider from 0 (geometric polygons) toward 1 (organic, biological shapes):

At 0: clean geometric Voronoi. As warp increases, the value noise distorts UV space, bending cell boundaries into organic curves. The warp doesn't change cell topology — same cells, same neighbors, just different shapes.

4. Multi-Layer Composition

Scale progression

float scale = density * pow(layerSpread, fl);   // fl = 0, 1, 2

With defaults density = 8, layerSpread = 2:

LayerScaleCells across screen
08~8
116~16
232~32

This is a geometric frequency cascade — each layer has layerSpread× more cells than the previous. Same principle as fBm octaves, but applied to Voronoi patterns instead of noise.

Layer weighting

float layerWeight = pow(layerMix, fl);
if (layer == 0) layerWeight = 1.0;
LayerWeight (layerMix = 0.5)
01.0 (forced)
10.5
20.25

Layer 0 is always full-strength. Subsequent layers contribute with exponentially decreasing weight. At layerMix = 0: only layer 0. At layerMix = 1: all three layers contribute equally.

The normalization color /= max(totalWeight, 0.001) divides by 1 + layerMix + layerMix², keeping brightness roughly constant regardless of layerMix.

Contrast and brightness

layerColor = clamp((layerColor - 0.5) * contrast + 0.5, 0.0, 1.0);
color *= brightness;
color *= tint.rgb;

The contrast transform is an affine map around the midpoint 0.5:

output = contrast × (input − 0.5) + 0.5

At contrast = 1: identity. At contrast > 1: values push away from mid-gray. At contrast < 1: values compress toward mid-gray.

Contrast is applied per layer before blending, while brightness is applied after blending. If you swapped them, high contrast on individual layers would clip before blending, losing information.

5. Edge Detection & Glow

smoothstep as a soft threshold

float edgeFactor = 1.0 - smoothstep(0.0, max(edgeWidth, 0.001), edgeDist);

Recall edgeDist = F2 − F1, which is 0 on cell boundaries and positive inside cells. The smoothstep(0, w, d) function returns:

           { 0                          if d ≤ 0
s(d) =     { 3(d/w)² − 2(d/w)³   if 0 < d < w
           { 1                          if d ≥ w

So 1 − smoothstep(...) is 1 on the boundary and falls to 0 over a distance of edgeWidth. A soft edge line with controllable width.

The glow layer

float glowRange = edgeWidth * (1.0 + edgeGlow * 4.0);
float glowFactor = (1.0 - smoothstep(0.0, glowRange, edgeDist)) * edgeGlow;
float totalEdge = clamp(max(edgeFactor, glowFactor), 0.0, 1.0);

The glow is a second, wider smoothstep threshold with range edgeWidth × (1 + 4·edgeGlow). At default edgeGlow = 0.4:

glowRange = edgeWidth × 2.6

The glow extends about 2.6× further than the hard edge. The max(edgeFactor, glowFactor) ensures the core edge is never dimmed by the glow. Visually: a bright core line with a softer halo around it.

Edge color

vec3 edgeRGB = hsv2rgb(vec3(hue, colorSat * 0.2, 1.0));

Same hue as the cell interior, but saturation dropped to 20% (closer to white) and brightness at full. Bright, pale edge lines that “glow” against the more saturated cell interiors.

Try adjusting both sliders to see the hard edge and soft glow independently:

edgeWidth controls the hard line thickness (the smoothstep threshold). edgeGlow adds a wider, dimmer halo. Set edgeWidth to 0 to see glow-only; set edgeGlow to 0 for hard lines only.
